# 🎣 Phishing Threat Detector

Local, rule-based triage for suspicious emails, with an optional plain-English
write-up generated by a local LLM (via [Ollama](https://ollama.com)). Built to
automate the first pass of phishing detection and reporting: upload an email,
get a risk score with evidence, and export an incident-ready report — all
without any data leaving your machine.

## What it does

- **Parses** `.eml` files or pasted email text — headers, plain/HTML body, links, attachments.
- **Extracts and scores URLs** for phishing indicators: raw-IP links, URL shorteners,
  disposable/free TLDs, punycode/IDN tricks, `@`-symbol tricks, excessive subdomains,
  typosquatting and brand-impersonation (via edit-distance and segment matching), and
  mismatched anchor-text-vs-href links in HTML mail.
- **Detects suspicious language** across six categories: credential harvesting, urgency,
  financial lures, threats, authority impersonation, and generic greetings.
- **Flags sender anomalies**: display-name brand impersonation, and From/Reply-To/Return-Path
  domain mismatches.
- **Flags risky attachments**: executable/script extensions, macro-enabled Office files,
  and double extensions (e.g. `invoice.pdf.exe`).
- **Scores everything into a 0–100 risk score** (Low / Medium / High / Critical) with a
  transparent, documented breakdown — no black box.
- **Translates findings into plain English** using your local Ollama model, for people who
  don't want to read a list of regex flags — and exports the whole thing as a Markdown
  incident report.

It ships with two synthetic sample emails (one phishing, one legitimate) so you can try it
immediately without needing a real phishing sample on hand.

## Why Streamlit (and how to use Flask instead)

The brief mentioned Streamlit or Flask — this build uses **Streamlit**, since it gets a
polished, interactive interface (file upload, tabs, live results, streaming AI output) built
and running with a fraction of the code Flask would need for the same UX.

All the actual logic lives in `modules/` and is plain Python with **zero Streamlit
imports** — `app.py` is the only file that touches the UI layer. If you'd rather have a
Flask app, you can wire up routes that call the same functions
(`email_parser.parse_eml_bytes()`, `url_analyzer.analyze_urls()`,
`keyword_detector.detect_keywords()`, `risk_scorer.score_email()`,
`llm_reporter.stream_report()`) and reuse 100% of the detection logic.

## Setup

**Prerequisites:** Python 3.9+, and [Ollama](https://ollama.com/download) installed locally
if you want the AI write-up feature (the risk scoring itself works without it).

```bash
# 1. Create and activate a virtual environment (recommended)
python3 -m venv venv
source venv/bin/activate      # Windows: venv\Scripts\activate

# 2. Install dependencies
pip install -r requirements.txt

# 3. (Optional, for the AI write-up) Start Ollama and pull a model
ollama serve
ollama pull llama3.2

# 4. Run the app
streamlit run app.py
```

The app opens at `http://localhost:8501`. If Ollama isn't running, everything except the
AI write-up still works — the sidebar will tell you how to start it.

## Project structure

```
phishing-detector/
├── app.py                     # Streamlit UI — the only file that imports streamlit
├── modules/
│   ├── email_parser.py        # .eml / raw text -> ParsedEmail (headers, body, URLs, attachments)
│   ├── url_analyzer.py        # Per-URL phishing indicator scoring
│   ├── keyword_detector.py    # Suspicious-language category matching
│   ├── risk_scorer.py         # Combines all signals into a 0-100 score + tier
│   └── llm_reporter.py        # Ollama integration: findings -> plain-English report
├── data/
│   ├── keywords.json          # Editable phrase lists by category, with weights
│   └── known_brands.json      # Editable brand -> official domain(s) map
├── sample_emails/
│   ├── phishing_example.eml   # Synthetic example for demoing/testing
│   └── legitimate_example.eml # Synthetic example for demoing/testing
├── .streamlit/config.toml     # Theme
└── requirements.txt
```

`data/keywords.json` and `data/known_brands.json` are plain JSON specifically so the
keyword lists and brand list can be extended without touching code.

## How the risk score works

The score is a transparent, weighted heuristic — not a trained classifier — so every
point is explainable:

| Signal | Max points | Based on |
|---|---|---|
| URLs | 35 | The single riskiest URL found in the email |
| Suspicious language | 30 | Sum of matched keyword-category weights |
| Sender headers | 20 | From/Reply-To/Return-Path mismatches, brand impersonation |
| Attachments | 15 | Dangerous, macro-enabled, or double-extension files |

**Tiers:** 0–19 Low · 20–44 Medium · 45–69 High · 70–100 Critical

The same breakdown that produces the score is what's shown in the UI and handed to the
LLM — the AI write-up is asked to *explain* these findings, not invent its own verdict.

## Design notes / limitations

- **No live requests are ever made to URLs found in a scanned email.** All URL analysis
  is lexical/structural (patterns in the URL string itself). This is a deliberate safety
  choice for a tool whose whole purpose is handling untrusted, attacker-authored content.
- **Email HTML is parsed, never rendered.** The app reads links and text out of HTML
  bodies with BeautifulSoup; it never displays the email's HTML as live HTML.
- **This is a heuristic triage aid, not a certified detection engine.** It's built to
  demonstrate the pipeline (extraction → detection → scoring → LLM explanation) clearly
  and to be genuinely useful for a first pass — not to replace enterprise email security
  or a trained analyst's judgment. Expect false positives on unusual-but-legitimate email
  and false negatives on novel phishing patterns not covered by the keyword/brand lists.
- **Domain parsing is a lightweight heuristic**, not a full public-suffix-list
  implementation, so a handful of uncommon multi-part TLDs may be handled slightly
  imprecisely.

## Possible extensions

- Live reputation lookups (e.g. VirusTotal, URLhaus) as an opt-in, clearly-labeled feature
- SPF/DKIM/DMARC header validation
- A trained classifier alongside the heuristic scorer, with the LLM explaining agreement/disagreement
- A Flask/FastAPI version exposing the same `modules/` pipeline as a REST API for SOC tooling integration
